# Feature Specification: 历史休假记录时点额度计算修复

**Feature Branch**: `002-historical-leave-balance`
**Created**: 2025-11-17
**Status**: Draft
**Input**: User description: "修复历史休假记录额度计算错误：添加2021年4月1号入职的员工，填写2024年9月的休假记录时，应该消耗2024年9月时点的额度，而不是当前时间的额度"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 正确计算历史休假记录的可用额度 (Priority: P1)

HR管理员为员工补录历史休假记录时（例如补录去年的休假数据），系统应该基于休假发生日期当时的年假额度来验证和扣减，而不是使用当前时间点的额度。

**Why this priority**: 这是核心功能缺陷修复。如果不修复，所有历史数据录入都会产生错误的额度计算，导致年假余额不准确，影响整个系统的数据完整性和可信度。

**Independent Test**: 可以通过创建一个2021年4月1日入职的员工，然后记录2024年9月的休假，验证系统是否使用2024年9月时点的年假额度（而非当前2025年11月的额度）来进行余额验证和扣减。

**Acceptance Scenarios**:

1. **Given** 系统中有员工"张三"，入职日期2021年4月1日（已在2021年10月1日获得10天、2022年10月1日获得11天、2023年10月1日获得12天、2024年10月1日获得14天、2025年10月1日获得16天），当前日期为2025年11月17日，**When** HR管理员记录张三在2024年9月15日的1天休假，**Then** 系统应该基于2024年9月15日时点的年假额度进行验证。在该时点，考虑2年有效期后的可用额度为：2022年10月1日获得的11天（2024年10月1日过期前有效）+ 2023年10月1日获得的12天 = 23天（2021年10月1日获得的10天已于2023年10月1日过期）。如果该时点张三未使用过任何年假，系统应允许记录1天休假，从即将过期的11天额度中扣减

2. **Given** 系统中有员工"李四"，入职日期2023年1月1日，当前日期为2025年11月17日，**When** HR管理员记录李四在2023年6月1日的休假（入职满5个月，当时还未获得首次年假），**Then** 系统应提示"该日期时点年假余额不足"或"该日期时点尚未获得年假额度"，阻止记录创建

3. **Given** 系统中有员工"王五"，入职日期2020年1月1日，在2022年12月有10天年假余额，**When** HR管理员记录王五在2022年12月31日的15天连续休假，**Then** 系统应提示"该日期时点年假余额不足（可用10天，需要15天）"，阻止记录创建

4. **Given** 系统中有员工"赵六"，入职日期2021年1月1日，**When** HR管理员记录赵六在2023年8月的3天休假，然后再记录2024年9月的5天休假，**Then** 两次记录都应基于各自日期时点的额度进行验证和扣减，且当前余额计算应考虑所有历史扣减

---

### User Story 2 - 历史休假记录对当前余额的正确影响 (Priority: P1)

补录的历史休假记录应该正确地反映在员工当前的年假余额中，确保历史扣减和当前额度计算的一致性。

**Why this priority**: 这是P1优先级，因为余额计算的准确性直接影响HR管理决策和员工权益。如果历史记录不能正确影响当前余额，系统将失去数据完整性。

**Independent Test**: 可以通过查看补录历史休假前后的员工年假余额变化，验证系统是否正确追溯计算。

**Acceptance Scenarios**:

1. **Given** 员工"张三"当前显示剩余12天年假，**When** HR管理员补录张三去年的5天休假记录，**Then** 系统重新计算后显示张三当前剩余7天年假

2. **Given** 员工"李四"当前显示剩余8天年假，**When** HR管理员补录李四2年前的3天休假（该部分年假已过期失效），**Then** 系统显示李四当前剩余8天年假不变（因为扣减的是已过期的额度）

3. **Given** 员工"王五"当前显示剩余15天年假，**When** HR管理员删除王五去年的一条4天历史休假记录，**Then** 系统重新计算后显示王五当前剩余19天年假

---

### User Story 3 - 跨年度历史记录的正确处理 (Priority: P2)

系统需要正确处理跨越多个年假发放周期的历史休假记录，确保在不同时间点获得的年假额度被正确消耗。

**Why this priority**: 这是P2优先级，是P1功能的自然延伸。虽然重要，但只要P1的时点额度计算正确，这个场景理论上应该自动支持。

**Independent Test**: 可以通过为一个长期在职员工补录多年的历史休假记录，验证系统是否正确处理不同年假发放批次的额度消耗。

**Acceptance Scenarios**:

1. **Given** 员工"张三"入职2020年1月1日，在2020年7月、2021年7月、2022年7月分别获得10天、11天、12天年假，**When** HR管理员补录张三在2021年12月的8天休假，**Then** 系统应按FIFO原则，优先扣减2020年7月获得的10天额度（剩余2天），再扣减2021年7月获得的11天额度中的6天（剩余5天）

2. **Given** 员工"李四"在2023年有多批年假额度（包括自动发放和手动调整），**When** HR管理员补录李四在2023年10月的休假，**Then** 系统按照"最早过期优先"原则扣减额度（自动发放的2年有效期额度优先，手动调整的永久有效额度最后）

---

### Edge Cases

- 如果HR管理员补录的历史休假日期早于员工入职日期，系统应如何处理？（答：系统应验证并阻止，提示"休假日期不能早于入职日期"）
- 如果HR管理员补录的历史休假日期晚于当前日期（未来日期），系统应如何处理？（答：允许记录，但给予提示"该日期为未来日期"，按未来时点的预期额度进行验证）
- 如果员工在某个历史时点的所有年假额度都已过期，补录该时点的休假应如何处理？（答：系统应允许记录，但标注"此休假使用的额度已过期"，不影响当前余额）
- 如果补录历史休假导致某个时点的余额为负，系统应如何处理？（答：系统应阻止记录，提示"该时点余额不足，可能存在数据冲突或遗漏"）
- 如果系统中已有该日期的休假记录，再次补录应如何处理？（答：按现有规则，阻止重复记录）
- 删除历史休假记录后，系统如何重新计算余额？（答：系统应重新计算所有历史时点和当前的余额，确保数据一致性）
- 如果员工在某个历史时点有手动调整的额度增加记录，补录更早时点的休假应如何处理？（答：按实际时间顺序重新计算，确保每个时点的额度状态正确）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在记录年假使用时，基于休假日期（而非当前日期）来计算该时点的可用年假额度
- **FR-002**: 系统必须在验证年假余额时，计算截至休假日期为止，员工应获得的所有年假额度（包括已发放的自动额度和手动调整额度）
- **FR-003**: 系统必须在验证年假余额时，计算截至休假日期为止，员工已使用的所有年假天数（基于该日期之前的所有使用记录）
- **FR-004**: 系统必须按照FIFO原则（先过期先使用），在历史时点正确分配年假扣减到相应的额度批次
- **FR-005**: 系统必须验证休假日期不能早于员工入职日期，如果违反则阻止记录创建并提示错误
- **FR-006**: 系统必须在添加、修改或删除任何年假使用记录后，重新计算所有受影响员工的当前年假余额
- **FR-007**: 系统必须在历史时点年假余额不足时，阻止休假记录创建并提示具体的余额不足信息（包括该时点可用天数和需要天数）
- **FR-008**: 系统必须在年假额度数据结构中记录每批额度在每个历史时点的使用情况，以支持时点查询
- **FR-009**: 系统必须正确处理跨越年假有效期的历史记录，确保过期额度在其有效期内被正确使用
- **FR-010**: 系统必须在补录历史休假时，如果该时点员工尚未获得任何年假额度（如入职不满6个月），则阻止记录并提示"该日期时点尚未获得年假额度"
- **FR-011**: 系统必须支持补录未来日期的休假记录，并基于该未来时点的预期额度进行验证（同时给予提示）
- **FR-012**: 系统必须在删除历史休假记录后，正确回滚该记录对各批次额度使用情况的影响
- **FR-013**: 系统必须在用户界面上清晰显示正在记录的是"历史休假"还是"当前休假"，以避免用户混淆

### Key Entities

- **LeaveEntitlement（年假额度）**: 需要增强以支持历史时点查询，关键属性包括：已有的所有属性，plus 按时间点追踪的使用历史记录
- **LeaveUsage（年假使用记录）**: 保持现有属性，date字段既是休假发生日期，也是计算余额的时点基准
- **时点余额快照（Point-in-Time Balance Snapshot）**: 概念性实体，用于表示某个特定日期时点的年假余额状态，关键属性包括：日期、总额度、已使用天数、剩余天数、各批次额度的分配情况

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: HR管理员能够正确补录任意历史日期的休假记录，系统基于该日期时点的额度进行验证，验证准确率100%
- **SC-002**: 补录历史休假记录后，员工当前年假余额的计算准确率达到100%，所有测试案例中余额变化符合预期
- **SC-003**: 系统能够在1秒内完成基于任意历史时点的年假余额计算和验证
- **SC-004**: 用户在补录历史休假时，能够清楚地看到该时点的可用余额信息，减少因余额不足导致的操作失败率50%
- **SC-005**: 删除历史休假记录后，系统能够在2秒内完成余额重新计算并更新UI显示
- **SC-006**: 系统正确处理100%的边界情况（如入职前、有效期外、余额不足等），并提供清晰的错误提示
- **SC-007**: 修复后，HR录入历史数据的时间效率提升60%（因为减少了余额验证错误导致的重试）

## Assumptions

1. 系统假设所有年假使用记录的日期字段（date）准确反映休假实际发生的日期
2. 系统假设历史休假记录的补录是HR管理员的合理需求，用于数据迁移或补录遗漏的记录
3. 系统假设FIFO（先过期先使用）原则在历史时点同样适用
4. 系统假设手动调整的年假额度在其创建日期后即对该时点及之后生效
5. 系统假设不需要支持修改已有休假记录的日期（只支持删除和重新创建）
6. 系统假设年假额度的发放日期计算逻辑（基于入职周年）保持不变
7. 系统假设时点余额计算需要考虑所有类型的额度来源（自动发放、手动调整）
8. 系统假设过期的年假额度在其有效期内可以被历史休假记录消耗
9. 系统假设补录大量历史数据时，性能应保持在可接受范围（每条记录1秒内完成）
10. 系统假设当前余额显示始终基于当前日期重新计算，不依赖缓存的历史状态

## Clarifications

### Session 2025-11-17

- Q: 对于2021年4月1日入职的员工"张三"，在2024年9月15日时点的可用年假额度应该如何计算？ → A: 需要考虑年假发放时间线和2年有效期。张三在2021年10月1日获得10天、2022年10月1日获得11天、2023年10月1日获得12天。在2024年9月15日时点，2021年获得的10天已于2023年10月1日过期，因此可用额度为2022年的11天+2023年的12天=23天（假设未使用过）
